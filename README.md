The Gaming Room project was commissioned by a company that looked to broaden its successful mobile game, "Draw It or Lose It," to a web-based platform. The game was conceptualized as a version of Pictionary where groups contended by identifying a sketch within a specific time limit. The company wanted to develop a software solution that would enable multiple teams and players to participate in the game at the same time fairly while also ensuring secure user sessions. Their main requirements were the ability to run concurrent users, ensure every game and player had a unique identifier, and give cross-platform accessibility and scalability for future development.

In the course of writing the software design document, I was particularly successful in drawing the system architecture and directly identifying the object-oriented nature of the application. I presented convincingly that classes like Game, Team, and Player would derive from a common abstract class Entity, hence ensuring the code would be reusable and maintainable. I also backed the logic for selecting specific technologies and design patterns, like the Singleton pattern which is used to manage one instance only of the game service and avoid data duplication during gameplay.

The process of creating the design document was indeed an enabler for the actual development of the code. The documentation was like a schematic design, so I was able to see the system interactions in my mind before I implemented them. It clarified the relations between the classes, the chance of logical errors was decreased, and the alignment between the client’s needs as well as the good practices of software engineering was preserved.

In case I would be implementing one part of the document, I would emphasize the need to make the interface design section better. Although I did mention how data would flow from the user interface to the backend, I should have presented clearer mock-ups or descriptions of the user experience. This improvement would not only help the document become more non-technical stakeholder-friendly but also foster communication among the development team.

While I was interpreting the user’s needs, I first took an overarching approach of converting their goals-cross-platform compatibility, fairness, and scalability into technical specifications. For example, I backed the design of the system that packaged the web deployment through a RESTful API, an API that would enable users to access the game from any platform with internet access. Considering end-user motivation is important because the software should not only run properly but also deliver a glitch-free and satisfying experience to both the client and the players, by the use of which the client immersed.

In the software design, I used an object-oriented and modular strategy and started with a requirements analysis, then created UML diagrams to visualize system relationships, and finally mapped those designs into class structures. Going forward, I would continue to use these structured design methods but mix them with agile development practices like iterative prototyping and continuous feedback loops. Instead of sticking to a rigid sequence of tasks, I would tailor the design dynamically as user needs shift.
