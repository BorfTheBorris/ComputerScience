Artemis Financial was a financial services company I contracted with to conduct a vulnerability assessment and to add secure communication to its web application. Specifically, the client wanted me to review the software for vulnerabilities that could lead to exposing their customer’s and financial data and to implement secure communication to protect that data. My contributions included properly identifying weaknesses as vulnerabilities using an automated tool for scanning and testing called vulnerability scanning and a manual review of the software’s source code. 

Identifying vulnerabilities through scanning that is not an actual weakness is known as a false positive. False positives and a few false negatives do not compromise your customer’s data, damage your business, and ruin your reputation can be ignored. Unfortunately, both the scanning and the review process identified a few weaknesses. I also added functional programming in the form of a service endpoint to get a SHA-256 checksum of the argument. The software was designed, implemented, and tested using secure coding at the software’s source. Secure coding is a first step in software security that can save your customer’s, your business, and your reputation.

The most useful part of the vulnerability assessment was the inclusion of the OWASP Dependency-Check tool as it clearly demonstrated that the easy to use and powerful, open-source Owasp Dependency-Check can check for vulnerable third-party components and dependencies. The most challenging part of the vulnerability assessment was deciding which of the results were actual weaknesses that could be exploited. I implemented HTTPS added in the form of an SSL connector in the software’s embedded Tomcat web container by adding the appropriate settings in the application's source code so that sensitive financial transactions and data are encrypted using a keystore located at `src/main/resources/artemis.jks` as the implementation of a java KeyStore.

I ensured the application was working and as secure as possible by running the application, testing the hash endpoint, verifying that HTTPS worked, and using the dependency-check report to check for vulnerabilities. After making changes based on the report, I reran the test and the report to ensure no new vulnerabilities were introduced. This solidified my understanding of the necessary resources and the approach to take when I wanted to develop applications that use checksums to guarantee the integrity of uploaded and downloaded files. The techniques and tools I leveraged while tackling this challenge include Maven and Spring Boot applications, OWASP’s Dependency-Check tool, Java’s MessageDigest class, knowledge of secure coding, and SSL. To impress a prospective employer I would share the vulnerability assessment report with them, show them the secure version of the code, and prove that running the dependency-check on the Gradle report produces no high vulnerabilities and that the checksum application works with HTTPS.
